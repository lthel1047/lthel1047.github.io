멕시코 DLC 할로윈 보스 전투를 개발하면서 판정 오차 문제를 겪었습니다. 초기 설계의 실수와 해결 과정을 기록합니다.

## 문제 상황

초기 설계에서 발사체(Projectile), 약점(WeakPoint) 판정까지 마스터가 모두 처리했습니다.

```
마스터: 발사체 생성 → 이동 → 충돌 판정 → 데미지 처리
게스트: 마스터의 발사체 위치만 따라가기
```

얼핏 보면 깔끔한 구조처럼 보였지만, 곧 문제가 드러났습니다.

## 발생한 버그

게스트가 패링을 했는데 데미지를 받는 현상이 반복됐습니다.

원인을 분석하면:

1. 게스트가 발사체를 패링함
2. 패링 결과를 마스터에게 전송
3. **그런데 마스터는 이미 "패링 안 됨 → 데미지"로 처리를 완료한 상태**
4. 네트워크 딜레이로 인해 판정 타이밍이 엇갈린 것

## 해결책 — 역할 분리

핵심 원칙을 바꿨습니다.

> **마스터는 핵심 상태 관리**, **클라이언트는 실시간 판정**

```csharp
// 발사체 생성은 마스터가 담당
// 하지만 생성 직후 소유권을 게스트에게 이전
[PunRPC]
void SpawnProjectileForGuest(int projectileId, Vector3 pos, Quaternion rot) {
    GameObject proj = PhotonNetwork.Instantiate(prefab, pos, rot);
    // 생성 후 소유권 이전 → 게스트가 직접 판정
    proj.GetComponent<PhotonView>().TransferOwnership(targetPlayer);
}
```

이렇게 하면:
- 발사체 **생성**은 마스터가 담당 (일관성 유지)
- 피격/패링 **판정**은 해당 클라이언트가 처리 (지연 없음)
- **결과값만** 마스터에게 전달

## 정리

| 역할 | 마스터 | 클라이언트 |
|---|---|---|
| 보스 상태 | ✅ 관리 | - |
| 발사체 생성 | ✅ 담당 | - |
| 피격/패링 판정 | ❌ 하지 않음 | ✅ 직접 처리 |
| 결과 전달 | ✅ 수신 | ✅ 송신 |

이 구조로 변경 후 판정 오차 문제가 완전히 해소됐습니다. 이후 Lusca 보스전에서는 처음부터 이 원칙을 적용해 안정적으로 개발할 수 있었습니다.
