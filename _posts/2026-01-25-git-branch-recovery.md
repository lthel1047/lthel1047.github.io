팀에서 브랜치 오염 사고가 발생했습니다. 선택적 머지 과정에서 대량의 파일이 누락된 것입니다. 복구 과정을 기록합니다.

## 사고 경위

A 브랜치에서 B 브랜치로 머지 작업을 진행했는데, 작업자가 필요한 파일만 선택해서 가져왔습니다. Git은 "나머지 파일들은 이미 머지된 것"으로 판단해버렸고, 결과적으로 대량의 파일이 누락됐습니다.

우리 팀의 근본적인 문제는 **뿌리가 되는 기둥 브랜치가 없다**는 것이었습니다. 오래된 이벤트 브랜치들이 살아있고, 수많은 머지와 cherry-pick이 쌓여 있었습니다.

## 시도한 방법들

### 1차 시도: revert 후 재머지

```bash
git revert <merge-commit-hash>
git merge <source-branch>
```

결과: 머지할 데이터가 없다고 나옴. revert가 "이미 머지된 상태"를 유지해버림.

### 2차 시도: revert의 revert

```bash
git revert <revert-commit-hash>
```

결과: 변함없음. 여전히 파일 누락 상태.

### 3차 시도: 이전 지점에서 새 브랜치 후 덮어쓰기

```bash
git checkout <pre-merge-commit>
git checkout -b recovery-branch
git merge <target-branch>
```

결과: 이미 너무 꼬인 상태라 충돌이 수습 불가 수준.

## 최종 해결책: 새 기둥 브랜치 생성

결국 **새로운 기둥 브랜치를 만드는 것**이 가장 깔끔한 해결책이었습니다.

1. 가장 최근의 정상 상태 커밋을 찾기
2. 그 시점에서 새 브랜치 생성
3. 이후 작업들을 cherry-pick으로 선별해서 가져오기
4. 팀 전체에 새 기둥 브랜치 공지

그리고 이 사고를 계기로 팀 Git 관리 가이드를 Notion에 작성해 공유했습니다.

## 예방 원칙

이 사고 이후 팀에서 정한 원칙들입니다.

- **절대 선택적 머지 금지** — 파일 선택 머지는 반드시 팀 리뷰 후 진행
- **기둥 브랜치는 항상 최신 유지** — develop 또는 main을 항상 배포 가능한 상태로
- **오래된 이벤트 브랜치는 주기적으로 정리**

사고는 언제든 일어납니다. 중요한 건 재발을 막는 프로세스를 만드는 것입니다.
